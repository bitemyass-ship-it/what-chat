| The system persists canonical chats from live events, but it still has no scheduled recovery path to reconcile missed chats or history after downtime, reconnects, or event loss. | Add a scheduled chat/message sync worker for analytics reliability. Introduce an internal cron or interval-based job that iterates available employee WhatsApp runtime sessions, refreshes chat lists with bounded concurrency, and incrementally backfills messages where needed. Persist sync progress and add tests for partial failure, deduplication, and recovery after missed live events. |
| Operators cannot manually refresh the chat list for one employee, so the UI depends entirely on previously ingested live events and cannot trigger an explicit backfill when opening an employee profile. | Add an on-demand employee chat sync flow. Introduce `POST /employees/:code/chats/sync`, expand the WhatsApp client abstraction to read chats, upsert the refreshed employee chat list into SQLite, and return the updated rows. Add tests for employees without runtime sessions, alias deduplication, and repeated idempotent sync runs. |
| The system has no way to backfill or refresh message history for a selected chat, so chat detail analytics cannot be populated from WhatsApp history and can only ever show messages captured after the feature is introduced. | Add chat-level message history sync and persistence. Create a `messages` table and repository, expose `GET /employees/:code/chats/:chatRecordId/messages` plus `POST /employees/:code/chats/:chatRecordId/messages/sync`, and use `getChatById`, `syncHistory`, and `fetchMessages` to backfill messages with deduplication by external WhatsApp message ID. Add tests for incoming/outgoing messages, repeated syncs, and pagination of stored history. |
| `PATCH /employees/:code` still assumes `employees.is_active` is the source of truth for session lifecycle, but `/employees/:code/whatsapp-session` can already start a runtime session for an employee whose persisted `is_active` is still `false`. A profile PATCH in that state can stop the runtime session unexpectedly, and rollback after a failed PATCH can skip restarting it because restart is still gated by `existingEmployee.isActive`. | Align employee update flow with the runtime-session model used by WhatsApp activation and delete. Decide whether PATCH should preserve runtime-only sessions for non-session field edits, and make rollback restart decisions based on pre-update `hasRuntimeSession` instead of persisted `employees.is_active`. Add tests for patching an inactive employee with an existing runtime session, for phone-number and display-name updates in that state, and for rollback after a failed PATCH. |
| The new `/employees/:code/whatsapp-session` activation flow can start a runtime session for an employee whose `is_active` flag is still `false`, while the existing update/delete flows stop sessions only when `employees.is_active` is `true`. This can leave orphaned runtime sessions alive after employee updates or deletion and cause chat persistence errors against a missing employee row. | Reconcile WhatsApp runtime activation with `employees.is_active` so there is a single source of truth for session lifecycle. Either persist activation state when starting `/whatsapp-session` or make update/delete flows consult runtime session state instead of only `is_active`. Add tests that cover activating an inactive employee, deactivating after runtime-only activation, and deleting an employee with an active runtime session. |
| A WhatsApp session started through `/employees/:code/whatsapp-session` is not durable across process restarts because bootstrap still restores sessions only from `employees.listActive()`. Users can complete QR login successfully and still appear disconnected after a server restart unless some other flow separately marks the employee as active. | Make WhatsApp activation durable across restarts by aligning bootstrap restore logic with the new activation flow. Either persist session-enabled state as part of `/whatsapp-session` activation or introduce an explicit persisted flag that bootstrap reads instead of relying only on `employees.is_active`. Add tests for restart behavior after successful QR activation. |
| One failed employee session can stop the whole service because batch startup uses `Promise.all()` and bootstrap exits on error. | Refactor WhatsApp session startup to use `Promise.allSettled()` so one failed employee login does not stop other active sessions or terminate the Express server. Add tests for partial startup success. |
| Disconnected or failed WhatsApp clients are removed from the session map but are not explicitly destroyed, which can leak Puppeteer/Chromium processes and listeners. | Extend the WhatsApp client abstraction with a cleanup method such as `destroy()` and ensure the manager calls it on initialization failure, disconnect, and shutdown. Add tests that verify cleanup is triggered. |
| Session persistence depends on the relative path `./sessions`, which can break across different working directories and look like a logout after restart. | Replace the relative LocalAuth path with an absolute configurable session directory derived from environment variables or project root. Add tests that verify the resolved path is stable. |
| After `auth_failure`, a broken client can remain registered as active and block re-initialization for that employee. | Update the manager so authentication failures mark the session as recoverable by removing the client from the active map and optionally retrying with backoff. Add tests for auth failure recovery. |
| The current log format for messages is inconsistent between the tested short format and the desired structured production format with `FROM`, `TEXT`, and `TIME`. | Standardize message logging behind the message handler so all incoming messages are emitted in a single structured format. Add tests that cover messages with and without timestamps and empty bodies. |
| Multi-session scale is linear in Chromium usage, which may become unstable as employee count grows. | Add operational safeguards for multiple WhatsApp sessions: document expected memory usage, cap concurrent sessions if needed, and add health metrics for active clients and disconnect rates. |
| Employee detail page keeps its own local form state but does not resync from fresh server props after `router.refresh()`, so the UI can stay stale after save or concurrent backend updates. | Update `EmployeeEditor` to treat refreshed server data as a source of truth: resync local state when `initialEmployee` changes or remount the editor on employee identity/version changes. Add tests that cover post-save refresh and concurrent backend updates. |
| The employee edit form still allows field edits while `PATCH` is in flight, so a slow save can overwrite newer user input with an older response payload. | Prevent save-time form races by disabling editable controls while saving or by ignoring stale responses that no longer match the latest draft. Add client tests that cover typing/toggling during an in-flight save. |
| A successful `PATCH` response with malformed JSON is currently surfaced as a generic network failure, which hides backend/proxy contract regressions and gives the user the wrong error state. | Handle invalid success payloads separately from transport failures in `EmployeeEditor` and employee proxy flows. Return a stable "invalid employee data" error for malformed `200 OK` responses and add tests for malformed JSON after successful status codes. |
| The new employee detail route and Next API proxy are not covered by request/integration tests, leaving regressions in `/employees/[code]`, `notFound()`, and GET/PATCH proxy semantics easy to miss. | Add frontend request/integration coverage for the employee detail page and `/api/employees/[code]` proxy, including happy path, 404 mapping to `notFound()`, backend error propagation, and body/content-type forwarding for `PATCH`. |

Add a small smoke check that `make dev` starts both processes or at least fails loudly instead of doing nothing. |
| The chat detail page renders "First Message At" and a full-history summary from a message list that is hard-limited to `10_000` rows by the backend, so long conversations can show incomplete history as if it were complete. | Introduce pagination or an explicit "latest N messages" contract for `GET /employees/:code/chats/:chatRecordId/messages`.
